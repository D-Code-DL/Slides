Final Webpage Project (Project #3)

Let's review the paper we are handing out.

The purpose of this website project is to create something that others can enjoy and that will bring of the skills you have gained as web developers together. You will be making a text adventure game!

A text adventure game is an interactive story that allows the user to select between choices that changes the course of the story.

This game should be fun, but it should also meet the sandards on the piece of paper that we will be handing out.

It is an individual project.

  • a two-paragraph explanation that describes why you chose the topic you chose, what it can be useful for, and any other information that you want to share with the user.
  • A step-by-step guide for implementing the thing that you are describing. This guide should include at least one code snippet to illustrate the topic. (You may add more if you want to show additional examples, or real use cases you have found.)
  • a quiz for users at the end, to answer at at least six questions about the information on your site
  • that requests the user's name
  • and includes one short text answer
  • that is scored
  • and provides the correct answers to the user
  • works cited (a list of your sources)

This project should be completed in pairs or on your own. If you work in a pair, the requirements are doubled (four-paragraph explanation, six code snippets, a twelve-question quiz).

Teamwork Makes the Dream Work

We’ve said this before and we’ll say it every time, “great software is created by great teams” -a bunch of smart people. We’ve also talked about the different roles on software teams: developers, quality assurance, designers, and project managers.

Building Great Teams

There’s science and studies about how to build a strong effective team. We’re just going to parse out some very important points:

  • Teams should create norms/agreements on how they will work together
  • Teams should have a common defined goal they all work towards
  • Teams should play to each other’s strengths and weaknesses
  • Teams should maintain honest and open communication

Keep In Mind

Before starting work, try to come up with some norms and define the goal. Communicate with one another about each other’s vision for the final product to ensure everyone’s on the same page. Remember that every member is valuable and necessary to the project.

Feedback Loops

In order for a team to work well together, teammates have to be open and honest with one another. This is especially important on software teams where there are many parts that have to move together effectively to produce a product. This is where the value of good feedback comes in.

Giving Constructive Feedback

When giving feedback, feedback should be:

  • actionable: A behavior that can be improved upon
  • specific: Feedback should be clearly defined
  • timely: Give feedback in the moment

Remember you’re giving feedback because you want to help, not harm.

Receiving Constructive Feedback

When receiving feedback, you should be:

  • receptive: Receive the feedback with an open mind
  • reflective: Reflect on what you’re being told and be open to the possibility that they’re right
  • inquisitive: Ask clarifying questions

Keep In Mind

While working on this project, remember the value of feedback. If a teammate’s actions are nudging the project off course, give them that feedback. If you’re receiving feedback from your teammates, remember it’s because they value you and the project.

Plan Ahead

Planning is one of the most important aspects of a successful project. Good planning can make a challenging project easier. Poor planning can make an easy project complicated. Proper project planning has multiple components such as, scope management, user story creation, estimating, project scheduling and resource management. Then after a plan is created it needs to be reviewed and revised throughout the project. How do you plan a software project?

Project Planning Overview

We aren’t going to take you through all the steps of project planning, only what we believe will be most vital to the success of this product (we’ll send out additional slides that covers the parts we’re skipping). Let’s start with something we’ve reviewed before: User Story Creation

Users Make the Product Go Round

Think about the product from the point of view of the user. This is important not just in development but also in design. Where should the button go? Well, where would the user expect it? Should we add a subheader? Well, would the user expect a subheader? Would a subheader help the user understand the information presented?

Break It Down

Now that you have your assignment. We want you to break down the project into smaller parts based on the user’s perspective. Ex: As a student user I want to read information about this javascript thing in paragraph form so that I can understand how it works.

Break It Down Again

After you’ve written your user stories, I want you to break them down into smaller tasks. The tasks should build up into the completed user story. Each task should be something one person/pair can work on. Ex:

  1. Research a javascript topic
  2. Outline the steps for the guide
  3. Draft a code snippet using the variables

Get Into the Flow

Development is a process. Planning, design, development, testing, repeat. For this project we want you to use this development cycle. Once you’ve planned your project and before writing any code, take a moment to think about and draft the design. Now you can start writing code to complete the tasks. After each user story, test it out to ensure the user is getting the planned experience. Repeat.
